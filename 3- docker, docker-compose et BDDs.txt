1-avec une VM, installer un serveur web : http://IP_du_serveur:80 

2-avec un conteneur docker il faut faire un mapping, "docker pull Nginx" puis "docker run -p 80:80 Nginx" (port_host=port_container)

-héberger une application 3 tiers : VM vs Containers

Client → Nginx → PHP → MySQL

1-avec une VM, installer le serveur web ayant php et installer MySQL, connecter le serveur web à MySQL, le serveur écoute sur 80 et 3306

2-avec dockers je créée 3 conteneurs "Nginx" "php:8.2-fpm" "MySQL:8" liés via un docker-compose.yml  (je n'expose que Nginx sur 80)

    Nginx utilise le port 80 (exposé)
    php:8.2-fpm utilise le port 9000 non exposé
    MySQL:8 utilise le port 3306 non exposé

docker-compose.yml
Nginx.conf -> lancer l'écoute sur 80
app/index.php ->créer une page qui se connecte à MySQL

lancer avec : docker-compose up -d
http://IP_du_serveur:80 -->la page index.php retourne : Connected to MySQL successfully!

Il y a trois façons pour lancer docker : 

  1-Faire un pull "docker pull nginx:latest" de limage puis l'exécuter avec "docker run -d -p 80:80 --name mon-nginx nginx:latest"

  2-Utiliser un dockerfile qui est une sorte de script : 
           1-docker build -t mon-image .  -->pour construire l'image en local   
           2-docker run -d -p 8080:8080 mon-image  --> pour lancer l'image 
       -->Ceci est utile si je veux appeler une image et executer aussi des instructions.

         Exemple : créer un fichier Dockerfile dans un dossier vide
         contenu : 
               # Dockerfile pour Nginx personnalisé 
                 FROM nginx:latest
               # Copier un fichier index.html personnalisé dans le dossier par défaut de nginx
                 COPY index.html /usr/share/nginx/html/index.html

         avoir un fichier index.html dans ce dossier

         docker build -t mon-nginx:1.0 .
         docker run -d -p 80:80 --name mon-nginx mon-nginx:1.0
                     
  3-Utiliser un docker-compose.yml pour orchestrer plusieurs conteneurs liés (frontend + Backend + BDD)
            docker compose build   # Étape 1 : build des images
            docker compose up      # Étape 2 : lancement des conteneurs
               en une seule ligne : docker compose up --build

           NB :  si le fichier docker-compose.yml appelle des dockerfiles, ils doivent être dans le dossier
                 si le fichier docker-compose.yml appelle uniquement des images dockerhub, rien n'est indisponsable.

Exemple de conteneurs : 

-Conteneur "service web" (API / site web) : docker run -d -p 8080:8080 my-api
  ex : backend Node.js, Flask, FastAPI, Spring Boot, Nginx…

-Conteneur interactif, terminal pour executer un script : docker run -it python:3.11

-Conteneur qui exécute un script unique et se ferme : docker run --rm -v "$PWD":/app -w /app python:3.11 python script.py

-Conteneur de base de données / file storage : docker run -d --name mysql-db -p 3306:3306 -e MYSQL_ROOT_PASSWORD=root -e MYSQL_DATABASE=ma_base  mysql:8     
                                               voir les fichiers de la BDD : docker exec -it mysql-db ls /var/lib/mysql  (mais stockage volatile)
          *Pour créer le conteneur BDD sur un dossier local (avec persistance des données) : 
                   docker run -d \
                     --name mysql-db \
                     -e MYSQL_ROOT_PASSWORD=secret \
                     -e MYSQL_DATABASE=ma_base \
                     -p 3306:3306 \
                     -v /home/user/mysql-data:/var/lib/mysql \
                     mysql:8

          pour accéder à mysql : docker exec -it mysql-db mysql -u root -p

Une image est un modèle statique pour créer un conteneur.
 ex, L'image node:20 contient tout ce qui est nécessaire pour exécuter une application Node.js sur un serveur Docker

Un conteneur est une instance en cours d'exécution d'une image Docker

liste des images de conteneurs : 
docker images 

-conteneurs en cours d'execution : 
docker ps

-conteneurs qui existent (démarrés ou arrêtés)
docker ps -a

++++ docker-compose.yml : orchestrer plusieurs services (conteneurs) connectés qui constituent une application.

-démarrer tous les conteneurs liés qui constituent une application : (app service on promise)
docker-compose up -d 
docker-compose down

-simple redémarrage : 
docker-compose restart

===redémarrage complet de l'image et ses données : 

docker-compose down --volumes   
docker-compose pull
docker-compose up -d

-démarrer ou arrêter un conteneur : 
docker stop container_ID
docker start container_ID 

docker rm : supprimer un conteneur
docker rmi : supprimer une image

├── docker-compose.yml
├── app/
│   ├── index.php

    └── init.sql

index.php est le fichier php qui se connecte à la BD et revoie du html et init.sql est le fichier qui crée la BD mysql et ajoute les données de départ.

--> j'aurai deux images dockers "php:8.2-apache" et "mariadb:10.5" qui peuvent charger deux conteneurs.

  une fois que c'est chargé, je peux modifier index.php et y ajouter du code, des pages ... pour améliorer mon site web 
                             ma base MySQL sera présente sur /var/lib/MySQL/BD que je peux exporter avec mysqldump "dump.sql" pour l'importer sur un serveur réel MYSQL en cas de besoin.

  mon application reste tout simplement le dossier "php-mysql-app"
  
point fort : j'utilise MYSQL et apache2-php sans avoir besoin de les installer réellement, docker me prépare l'environnement d'exécution.

-Les types de BDD :  

🔹 1. Bases de données relationnelles (SQL)
Organisées en tables avec des relations, elles utilisent le langage SQL.

Exemples :

MySQL / MariaDB (open source, très répandues)

PostgreSQL (plus avancée, supporte JSON, très robuste)

SQLite (léger, utilisé localement dans apps mobiles ou desktop)

Oracle Database (commerciale, entreprises)

Microsoft SQL Server (souvent dans l'écosystème Windows)

📌 Usage typique : systèmes de gestion, ERP, e-commerce, web apps classiques.

🔹 2. Bases de données NoSQL
Ne suivent pas forcément le modèle tabulaire, souvent utilisées pour la scalabilité et la flexibilité.

a. Clé-Valeur
🔸 Redis, Riak, DynamoDB

⚡ Ultra rapides, idéales pour du cache ou sessions.

b. Document
🔸 MongoDB, CouchDB, Firebase Firestore

📄 Stockent des documents JSON/BSON. Très flexibles.

c. Colonne
🔸 Apache Cassandra, HBase

📊 Optimisées pour les grandes quantités de données réparties (Big Data).

d. Graphes
🔸 Neo4j, ArangoDB

🔗 Stockent des nœuds et des relations. Idéal pour réseaux sociaux, recommandations, etc.

🔹 3. Bases de données orientées objets
🔸 db4o, ObjectDB

🧱 Stockent les objets tels quels (comme en Java).

🔹 4. Bases de données en mémoire
🔸 Redis, Memcached

⚡ Très rapides mais volatiles (sauf en mode persistant). Utilisées pour le caching.

🔹 5. Bases de données en temps réel / streaming
🔸 InfluxDB, TimescaleDB, Druid

📈 Utilisées pour les séries temporelles (capteurs, metrics serveur, IoT...).

🔹 6. Bases de données distribuées
🔸 Cassandra, CockroachDB, TiDB

🌍 Conçues pour la haute disponibilité et la tolérance aux pannes.
